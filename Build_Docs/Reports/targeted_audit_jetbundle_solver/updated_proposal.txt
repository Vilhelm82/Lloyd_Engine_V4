Why AlphaProbe comes before JetBundle
JetBundle without α becomes a typed AD bundle:
valuegradienthessianranksignature
Useful, but not uniquely V4.
The report confirms that α recovery is where V4’s moat is. The current Task 016 machinery can recover α−1 from transfer observations, but it is currently just fitted slope evidence over a supplied observation set. It is not yet a typed object that says:
this α was measured along this probefrom these samplesunder this precision pathwith these transfer traceswith this cancellation statuswith this stability behavior
That is exactly what AlphaProbe should provide.
So:
AlphaProbe = typed directional asymptotic exponent evidence.JetBundle = local measurement bundle that may include AlphaProbe evidence.Solver = typed navigator that consumes projection + AlphaProbe/JetBundle evidence.
No α, no proper Lloyd solver. We’d just be building Newton with receipts and a very handsome filing cabinet.

Alpha statuses
The report suggests a minimal set. I’d slightly expand it so future solver dispatch is clean:

But there’s a design choice here.
alpha_fractional, alpha_integer_degenerate, and alpha_negative are classifications of observed α. alpha_model_unique_match, alpha_model_ambiguous, and alpha_model_no_match are model-comparison statuses.
To avoid mixing two concepts too early, Task 018 could use:
alpha_observedalpha_model_unique_matchalpha_model_ambiguousalpha_model_no_matchalpha_unstable_windowalpha_cancellation_dominatedalpha_insufficient_dataalpha_domain_refusedalpha_nonfinitealpha_indeterminate
and store alpha_regime inside the value:
integer_positiveinteger_degeneratefractional_positivenegativezerounknown
But my preference is to let the status name carry the solver-relevant stratum:
alpha_regular_integeralpha_degenerate_integeralpha_fractional_branchalpha_negative_singularityalpha_non_algebraic_driftalpha_model_ambiguousalpha_model_no_matchalpha_cancellation_dominatedalpha_insufficient_dataalpha_domain_refusedalpha_nonfinitealpha_indeterminate
That makes the solver simpler later: it can dispatch on α-status directly.
Important acceptance rules
Task 018 must not do hidden model fitting magic.
It should use existing typed_finite_difference and typed_log_log_slope, but wrap them as first-class α evidence.
Rules:
1. f-values must be finite and positive.2. delta values must be finite and nonzero.3. Transfer observations must be typed results.4. Cancellation-dominated transfers are preserved but cannot support α acceptance.5. Slope is computed only from observed transfer evidence.6. observed_alpha = observed_slope + 1.7. declared_alpha_band is explicit caller evidence.8. Model matching uses declared band only.9. No hidden closeness-to-integer rule.10. Nested-window stability, if implemented, must use explicit windows and explicit stability band.11. AlphaProbe must never alter computed transfer values.12. AlphaProbe must preserve transfer and slope parent trace IDs.
That last cluster is the anti-goblin fence.

Updated minimal solver acceptance law
For the future solver, after AlphaProbe exists:
A candidate step may advance only if:1. Candidate evidence is protocol-valid.2. Projection evidence is defined and advance-valid.3. AlphaProbe evidence is observed or classified into an accepted alpha stratum.4. Candidate generator is compatible with that alpha stratum.5. No unhandled status transition occurs.6. Residual decrease may only break ties after typed geometry has permitted all tied candidates.7. If tied candidates remain indistinguishable under declared typed evidence, emit a typed tie-refusal.
That is the smallest proof that the solver is not residual descent in costume.
The first killer tests:
1. A residual-improving candidate is rejected because α evidence is unstable or incompatible.2. A slower candidate is accepted because it preserves projection and α evidence.3. Two candidates tied under projection and α evidence return typed candidate-tie refusal, not a hidden branch-order or residual-score choice.
