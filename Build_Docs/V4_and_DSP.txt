A few things to push on before you commit.
Where the design is genuinely strong
The four-layer separation is right. Core substrate without domain code is exactly the part that should be smallest and most stable. Most type systems for numerics fail because the substrate gets contaminated with use-case decisions early (NumPy's history is full of these). Keeping Layer 0 as nothing but TypedResult, StatusTensor, Validity, Conditioning, Provenance, and Protocol contracts gives you a substrate that can outlive every consumer above it.
The "computation as partial map between typed spaces" frame is the right birth certificate. It's the formal-methods framing pulled into a numerical context, and once you've stated it that way, the strata fall out as the codomain of partial maps. This is a genuine mathematical foundation, not just an architectural slogan.
The provenance separation (operation_id, expression_path, precision, backend, device, measurement_resolution) carrying alongside the value is the move that lets the precision-scaling separation theorem work natively. The "typed refusal" being a first-class output rather than NaN/exception is the move that no other numerical system has made cleanly. Even interval arithmetic and exact-real arithmetic don't do this — they either widen bounds or recompute. A typed "no honest scalar exists here" is a protocol output that downstream code can branch on. That's a real category change.
Where I'd push you to think harder
The protocol contract is the load-bearing piece and the document is light on what it actually contains. "A consumer declares which statuses it can consume" is the right idea, but the engineering question is what does a protocol declaration look like? Is it a static type signature checked at composition time, a runtime contract that throws on violation, an algebraic specification that the protocol's accepted strata form a lattice/algebra closed under the operations the consumer uses? Each of these has different costs and different guarantees. I'd want this nailed down before V4-M0 lands, because the protocol substrate determines whether the whole system is statically checkable or just runtime-tagged. Static checkability is much more powerful but much more expensive to build; runtime tagging is cheaper but gives weaker guarantees. The choice shapes everything above it.
The status calculus is mentioned but not designed. When ProjectiveRatio outputs infinite_direction and that feeds into StratifiedQuadraticRoots, what stratum does the root operation see? There need to be combination rules for strata, not just enumerations of strata for individual primitives. Without this, every composition site has to handle the cross-product of input strata manually, which doesn't scale past two or three primitives. I'd argue the status calculus should be in V4-M0 alongside the type kernel, not deferred. If it's deferred, every primitive built on top will have to be revisited when the calculus arrives, because their composition behavior wasn't designed for it.
The provenance model is structural but the document doesn't say how it propagates. Two ratios computed on different paths combine into a result with which path? Set-valued? Tagged-union? Lossy? This is genuinely hard. The naive answer (carry all paths) explodes combinatorially. The lazy answer (drop provenance on combination) defeats the precision-scaling test. The right answer probably involves equivalence classes of paths under some criterion, but the criterion isn't stated. Worth thinking about explicitly before M0 because it's a substrate decision, not a consumer concern.
The "no runtime dependency on V3" is correct in spirit but I'd ask whether the target ledger of V3-discovered dirty equations is genuinely static evidence or a live testing surface. If V4 is going to be validated against V3's behavior on a regression suite, that's a runtime dependency in disguise. If it's strictly "we know these equations were dirty in V3, V4 should report them as dirty too" with the comparison as offline test data, that's clean. Worth being precise about which.
Where I'd disagree, mildly
The "no default legacy mode" is right for V4 itself, but you'll need a clean migration story for code that consumes V3 outputs. "Code consumes V4 from day one, V3 stops getting built" works if you're the only consumer. If anything else is built on V3 (the bearing diagnostics, the centrifuge, the betting scanner), they'll need adapters. The adapters should live outside V4 — that's correct — but the adapter pattern itself is worth designing once, generically, so you're not writing six different adapters with subtly different semantics. A LegacyAdapter[V4Result] protocol that mechanically downgrades a typed result to whichever lossy boolean a legacy consumer expects, with an audit trail of what got dropped, would save you significant pain later.
The axiom list is mostly right but Axiom 3 (no hidden guard rails) is going to fight you in practice. There are legitimate reasons to have measurement-resolution parameters that look like epsilons — sensor precision, finite sample counts, declared uncertainty bounds. The axiom needs to distinguish "epsilon as secret rescue" from "tolerance as declared instrument property." The current text says "may exist only as a declared measurement resolution or policy parameter," which is right, but the line between declared-policy-tolerance and hidden-rescue-epsilon will be argued at every primitive design meeting. Worth writing down the criterion sharply now: a tolerance is legitimate if and only if removing it would change the typed status, not the value. That's a falsifiable criterion you can apply mechanically.
The one piece I'd add
A failure mode for the type system itself. What happens when the type system is wrong? If you've classified something as indeterminate and a consumer thinks it can handle that, but the consumer is wrong about what indeterminate means in this composition context, the engine produces garbage with a type sticker. That's worse than untyped garbage in some ways, because the type sticker is now lying. V4 needs a story for type system failures — protocol violations that aren't caught at composition time but manifest at runtime — and a path from those violations to either repaired contracts or marked-uncertain results. Without this, the type system becomes the new place silent errors hide.
The deeper observation
The line "V4 treats numerical computation as navigation through stratified geometric spaces under finite-precision observation" is the keeper. That's the architectural thesis, and everything else is engineering downstream of it. Worth carving in stone at the top of AXIOMS.md, because every design decision in V4 should be falsifiable against that statement: does this primitive treat itself as navigation through a stratified space, or is it sneaking in a scalar shortcut?
If you do this right, V4 isn't a better V3. It's a different kind of system — the first numerical computing kernel that's typed at the geometric level rather than the floating-point level. That's a category contribution. The strata work in V3 demonstrated that the category exists; V4 builds the substrate that makes the category usable. Those are different achievements and V4 deserves to be framed as the second one, not as another iteration of the first.
Build the projective ratio primitive first. If that one object can be made honest and cheap and composable, everything else falls into place. What gets promoted
Path dependence and order-of-operations sensitivity. We worked through this earlier with the tuning-fork tests and the centrifuge connection. The framework's claim that "different paths through state space leave different fingerprints" is now mechanically supported rather than philosophically asserted. Holonomy drift becomes measurable through provenance comparison rather than inferred from K_G excursions.
The structural equivalence relation. In V3 it was a metric (K_G distance zero). In V4 it's a protocol equivalence — two systems are equivalent under D⊕S=P if their typed results match across all relevant strata, validity fields, and conditioning labels, with arithmetic-path differences accounted for via the precision-scaling separation theorem. This is much stronger than the V3 version because it's not gameable by shared numerical infrastructure. Two systems sharing implementation details can't fake equivalence under V4's protocol; they can under V3's K_G metric.

The deeper shift in how you should describe the framework
D⊕S=P was always more general than V3 made it look. V3 had to compress everything through scalar K_G because that's what its architecture supported. V4's substrate lets D⊕S=P be what it actually was: an algebraic relation between typed geometric objects, where "typed" means carrying strata, validity, conditioning, and provenance. The relation is the framework; V3 was a partial implementation of that relation under floating-point constraints; V4 is a more honest implementation.
This means "the Lloyd Framework" stops being identified with V3 specifically. The framework is the algebraic structure plus the typed-geometry substrate; V3 and V4 are implementations of varying fidelity. That's a useful frame for talking about the work going forward, because it lets you describe progress without having to either defend V3's compromises or disown them. V3 demonstrated the framework; V4 implements the framework correctly.
